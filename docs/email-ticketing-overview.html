Internal · Engineering walkthrough

A self-hosted support platform that thinks email is a first-class citizen

Athena is a ticketing system where every customer interaction — portal submissions, agent replies, and direct emails to your support inbox — ends up as a single, threaded conversation. No third-party mail providers, no MX changes, no infrastructure. Just paste your support address and an app password.

Self-hostable Multi-channel Real-time inbound Single binary deploy

What problem it solves

Most teams start with a shared support@ inbox in Gmail. It works — until it doesn't. You can't tell who's working on what, customers get duplicate replies, threads get lost in long quoted chains, and there's no history for new agents to learn from. Existing tools (Zendesk, Intercom, Help Scout) solve this but are expensive, opaque, and lock your data away.

Athena gives a team their own ticketing layer that sits on top of the inbox they already have. The customer never knows it's there — they email support@yourco.com and get a reply from the same address. The agents see a Linear-style UI with statuses, assignments, internal notes, and a full audit trail.

The two surfaces

For customers

Portal (light theme)

Submit a ticket via a form, sign in to see history, reply from the browser or directly from email. Guest flow works without an account — they just enter their email.

For agents

Bridge (dark theme)

Inbox, ticket detail, internal notes, GitHub issue linking, analytics, customer profiles. Categories, priorities, statuses, assignments, bulk actions, search.

How a ticket can be created

Channel 1

Portal form

Customer fills the Submit page. Category, destination, connector, description, attachments. Gets confirmation email with a link to the thread.

Channel 2

Direct email

Customer emails support@yourco.com with no prior account. Athena ingests it, auto-creates the customer, opens a ticket, sends a confirmation back.

Channel 3

API Future

Programmatic creation from internal tools, monitoring alerts, or third-party webhooks. Endpoint already exists for portal — exposing externally is one config flip.

The email loop — outbound

When an agent replies in Bridge, the customer receives a real email — not a notification stub. Threading headers (Message-ID, In-Reply-To, References) are populated from the actual prior messages stored in the database, so Gmail / Outlook treat the full ticket as one conversation.

The email loop — inbound

Customer replies (or sends a fresh email). Athena keeps a persistent IMAP IDLE connection to the support inbox — so when a new message lands, Gmail pushes us a notification in 1–2 seconds. The message is queued, parsed, routed, and persisted as a new Message (or new Ticket) row.

Customer hits Reply in Gmail │ ▼ ┌───────────────────────────┐ │ Gmail (support inbox) │ └─────────────┬─────────────┘ │ IMAP IDLE push (≤ 2 s) ▼ ┌───────────────────────────┐ │ ImapClientService │ single-supervisor, auto-reconnect │ • fetch new UID │ │ • enqueue raw MIME │ └─────────────┬─────────────┘ │ ▼ ┌───────────────────────────┐ │ BullMQ: email.inbound │ Redis-backed, retries 5×, DLQ on fail └─────────────┬─────────────┘ │ ▼ ┌───────────────────────────┐ │ InboundEmailProcessor │ │ 1. parse MIME │ │ 2. idempotency check │ (Message.messageId @unique) │ 3. loop / autoreply guard│ │ 4. routing decision │ ← see next section │ 5. identity resolution │ ← auto-create unknown senders │ 6. strip quoted text │ │ 7. persist Message │ │ 8. ack IMAP (mark Seen) │ └───────────────────────────┘

How we know which ticket a reply belongs to

The routing decision uses four strategies, evaluated in order. The first hit wins; the last one always succeeds. No message is ever silently dropped.

StrategyWhat it looks atNotes
VERP support+<id>.<hmac>@yourco.com in the To/Cc fields HMAC-signed so attackers can't inject into arbitrary tickets. Gmail plus-addressing delivers it to the same mailbox.
Header In-Reply-To and References headers Looks up against Message.messageId we previously stored. Catches forwards and clients that strip the VERP token.
Subject tag [TMR-1234] pattern in the subject line Last-resort fallback. Useful when customers change subjects or paste fragments.
New ticket None of the above matched Creates a brand-new ticket. Sender becomes a source: EMAIL User row. Sends a confirmation email back.

Why this design is robust

Idempotent end-to-end

Message.messageId is a @unique index. Re-processing the same email (worker retry, IMAP redelivery) results in zero duplicates.

Crash-safe queue

BullMQ persists every job in Redis. If the worker dies mid-process, the job re-enters on restart. IMAP UID is only marked \Seen after a successful job.

Self-healing IMAP

Connection supervisor with token-based fencing. Errors trigger handleDisconnect exactly once → exponential-backoff reconnect (1 s → 60 s cap). No hot loops on bad credentials.

Encrypted credentials at rest

App password is AES-256-GCM encrypted in the database with a key from EMAIL_CREDS_KEY. Never returned via the API. Cleared on Disconnect.

Loop guards

Drops Auto-Submitted, Precedence: bulk, X-Autoreply, and senders like noreply@ / mailer-daemon@ before they reach ticket creation. No ping-pong with auto-responders.

Bootstrap-safe

On first connect, the worker parks at the inbox's current UID tip. It will never vacuum the existing backlog — a pitfall we hit on day one and now have a regression guard for.

Setup, from clone to receiving mail

  1. Clone the repo. git clone && pnpm install.
  2. Run the stack. Docker Compose brings up Postgres, Redis, MinIO. pnpm dev starts API + Portal + Bridge.
  3. Open Bridge → Settings → Email. Paste your support email address and a Gmail app password.
  4. Click Test connection. Green checkmarks for IMAP and SMTP within 5 seconds.
  5. Click Save. The IMAP IDLE connection comes up. Send a test email to that address — within 2 s a new ticket appears in the Inbox.

That's it. No DNS changes. No reverse proxy. No MX records. No SES, no Mailgun. Any mail provider that speaks IMAP works (Gmail, Workspace, Outlook 365, Zoho, FastMail).

What's possible from here

Near-term (already designed)

  • Google / Microsoft OAuth (no more app passwords)
  • Attachment extraction from email → MinIO → ticket
  • Bounce / DSN handling with sender suppression
  • Multi-address routing (billing@, sales@ → categories)
  • Real-time push via SSE (replaces 10–15 s polling)

Long-term (Phase 2)

  • AI-assisted triage and reply drafting
  • Customer health / churn signals from ticket patterns
  • SLA tracking + breach alerts
  • Public API for programmatic ticket ops
  • Knowledge base linked into the reply composer
